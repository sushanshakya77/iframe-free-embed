Encapsulation
without exile

Every widget you paste into someone else's page needs the same three things: styles that don't collide, globals that don't clash, and a host it doesn't break. For twenty years the answer was an <iframe> — a second document, a sealed room, your widget banished from the page it's supposed to belong to.

A Custom Element backed by Shadow DOM satisfies the same contract by scoping instead of banishing. Below are three plates that prove it, running live in this page right now. Open the devtools. Read the source. It's four files.

01

The boundary holds

Both cards below are built by the same function, from the same markup, with the same class names. The left one renders into the page. The right one renders into a shadow root. Flip the switch and watch this page's global stylesheet reach exactly one of them.
Light DOM unprotected
Shadow root scoped
Host stylesheet
/* global, unscoped, !important — a real merchant's theme */
p     { color: #ff2d1f !important; text-transform: uppercase !important; }
.card { display: none !important; }
.stars{ color: #ff2d1f !important; }
What just happened. Selectors don't cross a shadow boundary. .card { display: none !important } is as aggressive as CSS gets, and it still can't see inside the shadow root — not because the widget out-specified it, but because the host's selectors never match in there at all. No specificity war. No !important arms race. Different scope.
02

Theme crosses on purpose

The honest comparison. Both cards are <review-card>, registered by the same review-card.js. The only difference: the right one is loaded inside a real <iframe>. Turn the knobs.
Same document inherits
Inside an iframe sealed
Host CSS variables
/* what the knobs above are writing onto the host element */
#brand {
  --embed-bg: 17, 17, 20;
  --embed-text: 245, 245, 245;
  --embed-accent: 52, 211, 153;
  --embed-radius: 20px;
}
Why the left one moves and the right one doesn't. Inherited CSS custom properties pierce a shadow boundary by design — so encapsulation is the default and theming is opt-in. That's the exact inverse of the iframe, where isolation is total and theming is impossible. The framed card can't see these variables because variables don't cross documents, and no amount of CSS will change that. To theme it you'd rebuild the cascade by hand as a config API over postMessage.

And notice the height. That iframe is height="176" — a number hardcoded by hand. It has no intrinsic relationship to its content. A production embed measures its own scroll height inside the frame, postMessages the number out, and listens on the host to resize it — on every content change, every breakpoint, every late-loading image. The card on the left is just an element in normal flow.

        One thing the iframe still does better. A shadow root
        is an encapsulation boundary, not a security boundary —
        mode: "open" is reachable via
        element.shadowRoot, and closed only obscures
        the reference. It scopes style and DOM, not origin. If you have to
        execute untrusted third-party code — a user-authored plugin, an
        ad from a network you don't control — the iframe's
        sandbox and origin isolation are the correct and
        irreplaceable tool. Use one.
        


        But if you're rendering your own trusted UI over your own sanitized
        data, as the overwhelming majority of embeds do, the iframe is charging
        you for a guarantee you don't need. The question worth asking before you
        reach for one:
        am I isolating untrusted code, or am I just isolating my
          styles?
